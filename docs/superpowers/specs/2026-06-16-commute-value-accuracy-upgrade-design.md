# Commute-Value Finder — 정확도 업그레이드 설계

- **작성일:** 2026-06-16
- **대상:** `commute_value_finder/` 전체 파이프라인
- **목표:** "저평가 입지(Blue Zone)" 판정의 통계적 신뢰도를 높이고, 단지 단위 해상도와 더 정확한 통근/접근성 반영, LLM 브리핑 신뢰도까지 끌어올려 "쓸 만한 서비스" 수준으로 품질을 끌어올린다.

---

## 1. 배경 — 현재의 핵심 결함

원본 거래 데이터(`data/seoul_apt_transactions.csv`, 76,986건)에는 거래 건별로 **전용면적·층·건축년도·아파트명**이 모두 있다. 그러나 현재 파이프라인은:

1. `map_builder.py`의 `aggregate_by_dong`에서 **동별 평당가 평균 한 값으로 뭉갠다** → 단위 특성(면적·층·연령) 정보가 소실된다.
2. `zone_analyzer.py`가 **`평당가 ~ 통근시간` 단변량 선형회귀** 하나로 저평가를 판정한다.

이 때문에 "저평가(Blue) 잔차"가 **입지 가치가 아니라 구성효과(composition)에 오염**된다. 예: 소형·신축 위주 동은 평당가가 높게 나오는데, 이는 입지 프리미엄이 아니라 매물 구성 탓이다. 단변량 모델은 이를 구분하지 못해 R²가 낮고(코드에 R²<0.3 분기 존재), Blue Zone 판정이 노이즈에 가깝다.

추가로:
- 통근시간이 **자동차 운전시간 단독**이며, API 실패 시 직선거리÷0.3km/min의 거친 추정으로 대체된다.
- 공간 해상도가 **동 단위**라 동 내부 편차(역세권/학군 블록 차이)를 못 잡는다.
- LLM 브리핑이 문서상 "Claude"라고 적혀 있으나 실제 구현은 OpenAI `gpt-4o`이며, 모델 교체 시 파라미터 비호환으로 조용히 폴백될 위험이 있다.

## 2. 설계 원칙

### 2.1 피처별 해상도 분리 (decoupled resolution)
- **통근시간은 느리게 변하고, 입지 가격 디테일은 빠르게 변한다.**
- **통근시간** → **동(323개) 단위**로 계산·캐시. 같은 동 내 단지는 동의 통근시간을 공유한다.
- **가치 모델·저평가 판정·지하철 접근성** → **아파트 단지(5,658개) 단위**로 산출한다.
- 효과: 유료/제한 API 폭주를 막으면서 단지 단위 해상도를 얻는다.

### 2.2 비용 원칙 — "LLM만 유료"
| 구성요소 | 비용 |
|---|---|
| 실거래가 (MOLIT/공공데이터포털) | 무료 |
| 단지 지오코딩 (카카오 Local) | 무료 |
| 운전 통근시간 (카카오 모빌리티) | 무료 (월 30만건, ~323건) |
| 지하철 접근성 (`subway_stations.json`) | 무료 (보유 데이터) |
| LLM 브리핑 (OpenAI `gpt-5.4-mini`) | **유료 (유일한 예외)** |

> ODsay 대중교통 API는 제외한다. ODsay 운영정책상 **데이터 분석 목적의 대중교통 API 사용은 별도 협의 + 최소 2,000만원** 대상이므로 무료 설계에 맞지 않는다.

### 2.3 인터페이스 분리로 단계화
통근 계산을 `CommuteEstimator` 인터페이스 뒤에 숨긴다. Phase A 구현(운전+지하철 접근성)을 먼저 완성하고, 진짜 대중교통(GTFS 멀티모달)은 **같은 인터페이스의 드롭인 교체(Phase B, 선택)**로 남긴다. 모델·대시보드 코드는 교체 시 바뀌지 않는다.

### 2.4 포지셔닝 — "판정 오라클"이 아니라 "정직한 스크리닝 도구"
저평가 잔차에는 우리가 통제하지 못한 모든 요인(학군·소음·향·환금성·재건축 가능성 등)이 섞인다. 따라서 잔차가 음수라고 해서 "저평가"라 단정할 수 없으며, **"이유가 있어 싼 것"일 수 있다.** 본 제품은 이를 인정하고:
- 결과를 **"가격이 낮은 후보 — 이유 확인 필요"**로 제시한다(단정 금지).
- 각 후보에 **확인 체크리스트**(학군·향·소음·나홀로·재건축 등)와 **불확실성(거래 n·신뢰도)**을 함께 노출한다.
- 신호가 실제로 의미 있는지 **백테스트로 자기검증**한다(§7).
- 즉 사용자가 **방문 전 후보를 걸러주고, 왜 싼지 스스로 확인하도록 돕는** 탐색 도구로 포지셔닝한다.

```
                                                  ▼ ▼  ▼
   [T1] 1단계 헤도닉: log(평당가) ~ log(면적)+층+연령+연령²+C(거래년월)

```

### 5.0b `preprocessor.py` — 데이터 정제·검증 레이어 (신규)
수집(`data_collector`)과 모델(`value_model`) 사이에 정제를 분리(단일 책임).
- `거래금액` 문자열(`"84,500"`)→int 변환 **검증**, `전용면적` 0/비정상 제거.
- `해제여부=='O'`(취소거래) 제외.
- **중복 제거:** 동일 (아파트·법정동·층·면적·거래년월) 중복 행 제거.
- 평당가 극단치 **winsorizing/절단**(§5.4와 일관).
- 최근 N개월 필터, 구/동/아파트명 **표준화**(공백·표기 정규화).
- **데이터 품질 로그:** 제거 건수·사유, 결측, 거래건수 부족 단지 등을 `output/logs/`에 기록.

### 5.1 `commute.py` — 통근 추정 (Phase A)
- **인터페이스:** `CommuteEstimator.minutes(lat: float, lon: float) -> float`
- **Phase A 구현 `KakaoDrivingEstimator`:**
  - 카카오 모빌리티 길찾기 API로 (주거지 좌표 → 회사 좌표) **운전 소요 분** 계산.
  - **동 단위로 계산**(323회), `data/commute_cache.json` 캐시.
  - **fallback 개선:** API 실패 동은 직선거리÷고정속도 대신, 성공한 인근 동들의 (직선거리, 통근분) 관계로 보정 추정(간단 회귀/보간). 거친 0.3km/min 제거.
- **캐시·비용 방어(운영):** 좌표 dedup(동 단위라 기본 323회), 실패 시 **exponential backoff**, 원본 응답 JSON 보존(`data/raw/`), **일일 호출량 가드**, **`--refresh-commute` 플래그가 있을 때만 재조회**(평소엔 캐시만 사용). 카카오 쿼터는 개발자 콘솔에서 확인.
- **출발시각 가정 고정:** 통근시간은 단일 기준(예: 평일 출근 시간대)으로 가정·명시 — 일관된 비교를 위해 시간대를 고정한다.
- **Phase B(선택, 미구현):** 동일 인터페이스로 GTFS 멀티모달 구현체를 끼워 넣는다.

### 5.2 `subway_access.py` — 지하철 접근성
- 각 단지 좌표에서 **최근접 지하철역까지 도보 거리(km)** 계산 (haversine, `subway_stations.json`).
- 산출 컬럼: `subway_dist_km`(최근접 역 거리), 선택적으로 `nearest_station`(역명).
- 주의: `subway_stations.json`은 `line` 필드가 대부분 비어 있고 역 순서/인접 정보가 없다. **본 기능은 "거리"만 쓰므로 영향 없음**(라우팅 불필요).
- **좌표계·거리 정확도:** 모든 좌표는 WGS84(EPSG:4326)로 통일. 거리(지하철역·회사 직선거리·반경)는 위경도 그대로 계산하면 왜곡되므로 **haversine** 또는 **한국 미터 투영(EPSG:5186)** 으로 계산. Moran's I 공간가중치(§5.4)도 동일 좌표 기준 사용.

### 5.3 `apt_geocoder.py` — 단지 지오코딩
- 고유 `(구, 법정동, 아파트명)` 5,658개에 대해 카카오 **키워드 검색**으로 단지 좌표 확보.
- `data/apt_geocode_cache.json` 캐시(영구). 실패 단지는 동 좌표로 폴백.
- 무료 한도 내에서 1회성 수집(캐시 후 재호출 없음).

### 5.4 `value_model.py` — 2단계 헤도닉 모델 (핵심)

**전처리 / 견고화:**
- 평당가 `price_per_sqm = 거래금액 / 전용면적` (기존 컬럼 재사용).
- **취소거래 제외:** 수집기에서 받은 `해제여부`가 'O'인 거래(계약 취소)는 제거 → 허위 고가/저가 제거. (§5.0 참조)
- 이상치 제거: `price_per_sqm`의 1~99 백분위 밖 제거, 비정상 면적(≤0)·연령(<0) 제거.
- `building_age = 거래년도 - 건축년도`.
- **평형 구간(`area_band`):** 전용면적을 구간화(예: ~60 / 60~85 / 85~135 / 135~ ㎡)하여 평형 혼재 통제용 더미로 사용.

**1단계 — 구성효과 + 시점 제거(품질 보정):**
```
log(price_per_sqm) ~ log(area) + C(area_band) + floor + age + age² + C(거래년월)
```
- **위치 변수(동/단지)는 넣지 않는다** — 위치의 편차가 바로 우리가 측정하려는 신호이기 때문.
- **평형 정규화:** `log(area) + C(area_band)`로 평형별 단가차를 통제 → 소형/대형 혼재가 단지 지수를 왜곡하지 않게 한다.
- **시점 보정 강화:** `C(거래년월)`을 시장 가격지수로 추정하고, **모든 거래를 최신 월 기준으로 정규화**한다(낡은 거래가 가짜 저평가로 잡히는 것 차단). 월 지수는 별도 산출물로 저장(대시보드 노출).
- 거래 건별 잔차 `resid_i` = 단위특성·평형·시점을 제거한 가격 편차.
- 모델 R²·계수(면적/평형/층/연령/시점 효과)를 저장(대시보드 노출용).

**단지 집계 + 축소추정(shrinkage):**
- 단지별 평균 잔차 `r_c`와 거래건수 `n_c`.
- **Empirical Bayes 수축:** `r_c_shrunk = (n_c · r_c + k · r_dong) / (n_c + k)`
  - `r_dong` = 해당 동의 평균 잔차, `k` = 수축 강도(설정값, 예 5).
  - 거래 적은 단지일수록 동 평균 쪽으로 당겨 노이즈 억제(원본의 "2건 동등가중" 결함 해결).
- 결과 = **입지가치지수**(단지 단위).

**2단계 — 통근·접근성 대비 저평가:**
```
입지가치지수 ~ commute_minutes(동) + subway_dist_km(단지)
```
- 최종 잔차 `final_resid` = 통근 편의성과 지하철 접근성으로 설명되지 않는 가격 편차.
- `sigma = final_resid.std()` 기준:
  - **Blue (저평가):** `final_resid < -ZONE_SIGMA·sigma`
  - **Gray (적정):** `|final_resid| ≤ ZONE_SIGMA·sigma`
  - **Red (프리미엄):** `final_resid > +ZONE_SIGMA·sigma`
- **Blue 신뢰 게이트:** Blue로 확정하려면 잔차 조건 외에 **거래건수 ≥ `MIN_TRANSACTIONS_PER_COMPLEX`** + **최근성(최근 `RECENCY_MONTHS` 내 거래 존재)** + **confidence 충분**을 함께 만족해야 한다. 즉 "싸다"가 아니라 **"저평가 가능성이 높고 표본 신뢰도도 충분한 곳"**으로 정의. 게이트 미달은 Gray로 강등하고 "표본부족/오래됨"으로 표시.
- 단지 단위 zone을 동 단위로 롤업: **동 대표값 = 동 내 단지들의 거래건수 가중 평균 `final_resid`** → 동 단위 sigma 기준으로 동 zone 분류(지도 개요/하위 호환용).

**공간 자기상관 점검 (OVB 군집 아티팩트 방지):**
- 최종 잔차의 공간 자기상관을 **Moran's I**로 검정(`libpysal` 공간가중치 + `esda`). 인접 단지/동의 잔차가 서로 상관되면, Blue 군집이 진짜 미스프라이싱이 아니라 OLS 아티팩트일 수 있다.
- 유의한 양의 자기상관이면: (a) **검정 결과를 산출물·대시보드에 보고**(최소 요건), (b) 선택적으로 **spatial lag/error 모형**(`spreg`)으로 2단계를 보정해 잔차를 정제. (기존 GeoPandas 사용과 자연스럽게 연결)
- 의존성 추가: `libpysal`, `esda`, (선택)`spreg` — PySAL 계열.

**출력 산출물(CSV):**
- 단지 단위: `구, 법정동, 아파트명, lat, lon, 입지가치지수, commute_minutes, subway_dist_km, final_resid, deviation_pct, zone, transaction_count, confidence`
  - `confidence` = 거래건수 기반 신뢰도 등급(예: n≥10 높음 / 5~9 보통 / <5 낮음). §5.8에서 노출.
- 동 롤업: 기존 `dong_with_commute.csv` 호환 컬럼 유지(하위 호환).

### 5.5 `llm_briefing.py` — LLM 브리핑 (T4)
- **LLM 공급자: OpenAI로 통일.** `config.LLM_MODEL = "gpt-5.4-mini"`. (실제 `.env`에 `OPENAI_API_KEY` 존재, `ANTHROPIC_API_KEY` 없음 → OpenAI가 실제 상태와 일치)
- **파라미터 호환:** GPT-5 계열은 `max_tokens` 대신 `max_completion_tokens` 사용, `temperature`는 기본값만 허용될 수 있음 → 모델군에 맞게 호출 파라미터 조정.
- **폴백 가시화:** API 실패 시 조용히 규칙기반으로 넘어가지 않고, 실패 사유를 명시적으로 로그/표시.
- **출력 스키마 강제(할루시네이션 방지):** 자유 텍스트가 아니라 **구조화 JSON**(OpenAI structured outputs/JSON mode)으로 받는다 — `recommendations[].{rank, 구, 법정동, 아파트명, reason{가격·통근·거래신뢰도·주의}, avg_price_per_sqm, commute_min, residual, risk_notes[]}, disclaimer`. JSON 검증 후 사용자용 텍스트로 렌더.
- **프롬프트 규칙:** 데이터에 없는 동/단지 추천 금지, 수치는 입력에서만 인용, 거래건수 적으면 반드시 `risk_notes`에 표기, 이유를 가격/통근/거래신뢰도/주의점으로 구분.
- **프롬프트 캐싱(선택):** 반복되는 데이터 컨텍스트에 OpenAI prompt caching을 적용해 토큰 비용 절감.
- docstring의 "Claude API" → "OpenAI" 정정.
- **문서 정합성 (CLAUDE.md):** 3중 불일치를 OpenAI로 통일 — Tech Stack `Anthropic Claude API` → `OpenAI API (gpt-5.4-mini)`, env 섹션 `ANTHROPIC_API_KEY` → `OPENAI_API_KEY`, Architecture의 `GPT-4o` → `gpt-5.4-mini`.

### 5.6 `app.py` / `map_builder.py` — 대시보드·지도 (T5)

### 5.8 정직성·불확실성·행동 레이어 (`app.py` 중심)
모델 산출물을 매수자가 **오해 없이, 행동 가능하게** 받도록 가공한다.
- **재프레이밍:** 내부 라벨은 Blue/Gray/Red 유지하되, UI 표기는 **"저평가 후보 (확인 필요)"** 등 단정하지 않는 문구로.
- **확인 체크리스트:** 각 후보 단지에 방문 전 점검 항목 노출 — 학군 / 향·동 / 소음(도로·철도) / 나홀로·세대수(환금성) / 재건축 가능성 / 향후 공급.
- **리스크 자동 플래그:** 보유 데이터로 "왜 싸 보이는지"를 자동 표기해 맹신 방지 — 노후(연식 높음), 거래 희박(`n` 낮음), 소규모/나홀로(거래 단지규모 추정 작음), 가격 변동성 큼 등. (인구감소 등 외부요인은 체크리스트로 안내)
- **불확실성 표시:** 단지별 거래건수 `n`과 `confidence` 등급을 함께 노출. 낮음 등급은 "표본 부족 — 신중" 경고.
- **행동 링크아웃:** 각 후보에 **단지명 기반 네이버부동산·호갱노노 검색 링크**(현재 호가·매물 확인) + **단지별 최근 거래내역·평형별 분포** 표 노출.
- **시점 안내:** 데이터 기준 기간과 "현재 시점 정규화 적용됨"을 명시.

**actionable 후보 랭킹 (Zone 색칠 보완):**
- 사용자의 "**예산 B · 최대통근 T · 최소평형 P면 어디 봐?**"에 바로 답하는 **랭킹 뷰** 제공.
- 입력: 예산(매매/전세), 최대 통근시간, **최소 전용면적(평형)**, 우선순위.
- 출력: 조건을 만족하는 단지를 **저평가 점수순 정렬** + 리스크 플래그·confidence 동시 표기. (현재 대시보드 필터를 "지도 색칠"에서 "행동 가능한 후보 리스트"로 확장)

### 5.9 교통 호재 레이어 (`data/transit_projects.json` + 지도/플래그)
- GTX-A/B/C + 주요 신설노선(신안산선·위례신사선·동북선 등) **예정역 위치를 정적 JSON으로 관리**(`name, lat, lon, line, 개통예정연도`).
- **모델 피처로 넣지 않는다** — 호재를 통제(control)하면 "쌀 이유"로 신호를 흡수해 버린다. 대신 **Blue 후보 위에 얹는 설명 레이어**로 사용:
  - 단지별 **"최근접 호재역 거리 + 개통예정"** 플래그.
  - 지도 **오버레이**(예정역 마커, 토글).
  - 해석 강조: "통근 대비 저평가 **+ 교통호재 인접 = 강한 기회 신호**" / 반대로 호재 없는 노후 저평가는 리스크 플래그와 함께.
- 한계: 노선·개통시점은 **수작업 갱신(정적)**, 확정성 변동 가능 → UI에 "예정" 명시.

## 6. 구현 순서 (체크포인트마다 검증)

각 단계는 독립적으로 동작하며, 중간에 멈춰도 파이프라인은 항상 실행 가능하다.

1. **데이터 정합성·정제** (`data_collector.py`+`preprocessor.py`) — 취소거래 필드, 정제·검증·중복제거·품질로그 분리.
2. **T1 헤도닉 모델** (`value_model.py`) — 평형 정규화 + 시점 보정 + 축소추정 + Blue 신뢰게이트 + 공간자기상관. 기존 동 단위 통근으로 먼저 완성·검증. 가장 큰 정확도 레버, 외부 의존 0.
3. **통근 인터페이스 분리 + 캐시·비용 방어** (`commute.py`).
4. **지하철 접근성 피처** (`subway_access.py`) → 2단계 회귀에 추가.
5. **단지 해상도** (`apt_geocoder.py`) → 위치 단위를 동→단지로 격상.
6. **백테스트 검증** (`validation.py`) — 신호가 실제로 의미 있는지 자기검증(§7).
7. **LLM** (`llm_briefing.py`) — gpt-5.4-mini + 출력 JSON 스키마 + CLAUDE.md 문서 정합성/가이드.
8. **교통 호재 데이터** (`data/transit_projects.json`) — GTX·신설노선 예정역 정적 데이터 작성.
9. **대시보드 + 정직성·행동 레이어** (`app.py`, `map_builder.py`) — 새 산출물 + 체크리스트·불확실성·리스크플래그·링크아웃 + 랭킹 뷰 + 호재 오버레이 + 캐싱.
10. **테스트** (`tests/`) — 핵심 로직 단위 테스트(§8.1). 각 단계와 병행 권장.

## 7. 검증 기준

**백테스트 (신호의 의미 검증 — 핵심):** `validation.py`
- 앞 `BACKTEST_TRAIN_MONTHS`(9개월)로 모델 학습 → 단지 zone 산출.
- 뒤 3개월 거래로 **out-of-sample 실현가**를 비교: "Blue(저평가 후보)" 단지의 이후 거래가 모델 예측 대비 더 높게(=실제로 저평가였음) 형성됐는지 측정.
- 지표: Blue vs Gray vs Red 그룹의 **이후 실현 잔차(또는 상승률) 차이**. Blue가 유의하게 우위면 신호 유효, 아니면 "예측력 없음"으로 정직하게 보고하고 라벨 신뢰도를 낮춘다.

**기타 검증:**
- **Moran's I 보고** — 최종 잔차의 공간 자기상관을 검정해, Blue 군집이 OVB 아티팩트가 아닌지 점검(§5.4).
- 새 다변량 모델의 R²가 기존 단변량 R² 대비 유의하게 상승하는지 비교 출력.
- Blue 단지 상위 사례를 수동 sanity check(실제로 통근·접근성 대비 저렴한지, 명백한 disamenity가 보이는지).
- 기존 보유 데이터(`seoul_apt_transactions.csv`)로 **엔드투엔드 실행 성공**(API 키 없이도 캐시/폴백으로 완주).
- LLM 경로: 모델 교체 후 실제 호출 성공 + 폴백 발생 시 명시 로그 확인.

## 9. 범위 밖 (Non-goals)

- 진짜 대중교통 소요시간(GTFS 멀티모달, 버스 포함) — Phase B 선택 사항으로 보류(인터페이스만 준비).
